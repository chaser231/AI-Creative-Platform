Operating Model

Главная идея этого плана: не строить редактор как один монолитный поток задач, а разделить его на параллельные workstreams, которые сходятся на shared contracts и integration milestones.

Stream A

Frontend Editor Core

Canvas engine, interaction model, editor layout, panels, toolbar, format UX, AI entrypoints.

Stream B

Backend Platform

Persistence, assets, templates, exports, queues, provider gateway, audit, observability.

Stream C

Shared Contracts

Canvas schema, layer contracts, slot taxonomy, multi-format model, action vocabulary.

Stream D

Research and Innovation

Smart resize intelligence, scene graph, explainability, quality metrics, agent planning.

Ключевой принцип синхронизации Frontend не ждёт “готовый backend”, а backend не ждёт “готовый UI”. Обе стороны работают против общих схем: CanvasDocument, Layer, Template, FormatInstance, ResizeMapping, AIAction.
Mock-first editor Contract-first APIs Feature flags for risky scope Milestone-based integration Experimental tracks in isolation

Roadmap Phases

Последовательность фаз построена так, чтобы сначала дойти до полезного editor MVP, затем нарастить structured composition и multi-format, и только после этого поднимать truly smart функции вроде resize intelligence и agent mode.

Phase 0. Foundation and Contracts

2–3 недели

Заморозка доменной модели, app shell, storage strategy, contract fixtures и feature-flag matrix.

Frontend

  • App shell редактора
  • Mock store и canvas fixtures
  • Interaction state model

Backend

  • Domain model и API contracts
  • Storage architecture
  • Queue foundation

Output

  • CanvasDocument v1
  • Template v1
  • AIAction draft

Backend

Exit

Frontend / Backend Sync

Если frontend уходит глубоко в редактор, backend всё равно может параллельно строить второй этаж продукта. Это не “подготовка на потом”, а независимая работа над будущими точками силы платформы.

Пока frontend строит editor

Backend может делать

  • Persistence model, save/load, autosave
  • Asset upload и media processing
  • Version snapshots и diff payloads
  • Template catalog и thumbnail pipeline
  • Export jobs и artifact storage
  • AI provider gateway, moderation, cost tracking
  • Action registry, audit trail, observability
Пока backend строит платформу

Frontend может делать

  • Canvas engine и interaction model
  • Tool system и properties UI
  • Frames, templates, multi-format UX
  • Mock-driven AI entrypoints и generation UX
  • Resize review surfaces и agent approval flows
  • Keyboard editing и performance polish
Практический механизм Каждая milestone заканчивается заменой mock adapter на реальный backend capability без переделки UI-модели. Это работает только если contracts frozen достаточно рано.

Advanced Scope

Эти направления не стоит отталкивать “на потом”. Лучше рано заложить data model и contracts, даже если часть работы пойдёт в research и feature-flagged delivery.

Track A

Smart Resize Intelligence

  • Semantic slot taxonomy вместо purely geometric resize
  • Rule engine и human-readable constraints
  • Confidence scoring per layer and document
  • Alternative layout proposals
  • Learning loop по accepted/rejected mappings
Track B

Agentic Canvas Operations

  • Scene graph / canvas AST
  • Action vocabulary и tool preconditions
  • Plan-preview-approve architecture
  • Brand constraints и policy engine
  • Explainable trace и diff before apply
Почему это важно Если не заложить семантику и action model рано, smart resize станет набором хрупких эвристик, а AI agent будет привязан к UI-хаку вместо устойчивого языка действий.
