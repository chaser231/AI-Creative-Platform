Подсистемы канваса

Ниже собраны основные рабочие блоки редактора: объектная модель, панели, resize, взаимодействия на холсте и окружение проекта.

Слойная модель

На канвасе есть `text`, `rectangle`, `image`, `badge`, `frame`. Все слои получают общую геометрию, видимость, блокировку, opacity, constraints и template slot role.

Мастер / инстанс

Контент живёт в master component, локальный layout живёт в instance конкретного resize. При переключении resize канвас пересобирается из instance props.

Layers Panel

Дерево frame hierarchy, inline rename, visibility, delete, контекстное меню, multi-select и drag-and-drop внутрь frame.

Properties Panel

Свйоства артборда без selection, полный инспектор для single selection и type-specific свойства для текста, shape, image, badge и frame.

Панель форматов

Master resize, custom sizes, rename/remove, linked/unlinked content cascade, smart resize и синхронизация слоев на активный формат.

Взаимодействия на канвасе

Selection, marquee (выделение рамкой), deep-select внутри frame, drag, resize, snap guides, clipping rules, pan/zoom, измерение расстояний через Alt и file drag-and-drop.

Слои и их свойства

Все типы слоёв разделяют базовые поля геометрии и состояния, а сверху получают type-specific свойства. Для компактности детали сведены в раскрывающиеся блоки.

Общие поля любого слоя базовая модель слоя
  • Геометрия: `x`, `y`, `width`, `height`, `rotation`
  • Состояние: `visible`, `locked`, `opacity`
  • Связь с шаблоном: `slotId`, `masterId`
  • Поведение в resize: `constraints.horizontal`, `constraints.vertical`
  • Контекст auto-layout: `layoutSizingWidth`, `layoutSizingHeight`, `isAbsolutePositioned`
  • Нюанс инстанса: `detachedSizeSync` для отключения синхронизации размера image в instance-форматах
Text типографика и контейнер
Контент `text`, typography (`fontSize`, `fontFamily`, `fontWeight`, `letterSpacing`, `lineHeight`), цвет и opacity, align, `textAdjust = auto_width / auto_height / fixed`, `truncateText`, `verticalTrim`, `textTransform`. Есть inline editor по double click.
Rectangle заливка и обводка
`fill`, `fillEnabled`, `stroke`, `strokeEnabled`, `strokeWidth`, `cornerRadius`, `opacity`.
Image источник и fit-режимы
`src`, `objectFit = cover / contain / fill / crop`, width/height, opacity. В properties есть замена файла и переключение fit-mode, а в resize instances может отключаться sync размеров.
Badge лейбл и форма
`label`, `shape = pill / rectangle / circle`, `fill`, `fillEnabled`, `textColor`, `fontSize`, `opacity`.
Frame вложенность и auto-layout
`childIds`, `clipContent`, `layoutMode`, paddings, spacing, primary/counter axis alignment, sizing modes, style props и `groupSlotId` для совместной AI-генерации вложенных текстов.

Auto-layout, вложенные frame и corner cases

Frame в текущем редакторе работает не просто как прямоугольный контейнер. Он может хранить детей через `childIds`, включать clipping, раскладывать слои по auto-layout и сам быть дочерним элементом другого frame. Это одна из самых “системных” подсистем канваса, потому что она влияет и на ручное редактирование, и на шаблоны, и на smart resize.

Как устроен auto-layout

Auto-layout включается через `layoutMode = horizontal | vertical`. У frame задаются paddings, spacing, primary/counter axis alignment и sizing modes, а у дочерних слоёв учитываются `layoutSizingWidth`, `layoutSizingHeight` и `isAbsolutePositioned`.

Как считается layout

Layout engine идёт снизу вверх: сначала считает внутренние frame, потом внешние. Для nested frame делается два прохода, чтобы сначала стабилизировать размеры, а затем доуточнить позиции после пересчёта родительских контейнеров.

Кто участвует в flow

В auto-layout участвуют только видимые дети без `isAbsolutePositioned`. Absolute-positioned слои исключаются из flow, но остаются связанными с frame и двигаются каскадно вместе с ним.

`fixed`, `fill`, `hug`

Для детей используются `fixed` и `fill`, а для frame ещё и `hug`. `fill` делит свободное место по оси, `hug` вычисляет размер контейнера по дочерним слоям. Для текста с `auto_height` высота может быть пересчитана повторно, если ширина изменилась через `fill`.

Аспект Как работает сейчас Что важно понимать
Вложенность frame Структура хранится через `frame.childIds`, а для шаблонов дополнительно сериализуется в `layerTree`. Runtime-модель редактора остаётся плоской: вложенность задаётся ссылками, а не живым деревом объектов.
Клик по дочернему элементу Без deep-select клик по child layer внутри frame выбирает родительский frame. Для точечного выбора внутреннего слоя нужен Cmd/Ctrl или специальный deep-select flow.
Drag дочернего слоя Без deep-select drag может быть делегирован родительскому frame. Это защищает композицию контейнера, но может ощущаться как “двигается не тот объект”.
Drop в frame При drop внутрь frame слой переносится в `childIds`; для auto-layout frame вычисляется `dropIndex` по позиции относительно соседей. Для обычного frame важен сам факт вложения, а для auto-layout frame ещё и порядок дочерних элементов.
Clipping Если включён `clipContent`, клики, drag-start и marquee selection за пределами frame блокируются. Clip работает и на рендер, и на интеракции, поэтому слой может существовать внутри frame, но перестать быть доступным для клика вне clip bounds.
Resize frame У non-auto-layout frame дети пересчитываются по constraints. У auto-layout frame они переукладываются через layout engine. Поведение зависит не только от типа слоя, но и от режима родительского frame.
Принципы вложенности frame runtime + interaction model
  • Frame может содержать любые слои, включая другие frame.
  • Все слои лежат в одном store, а вложенность определяется через `childIds`.
  • Top-level render исключает детей frame из корневого списка, чтобы они не рендерились дважды.
  • Auto-layout frame считаются bottom-up: сначала более глубокие контейнеры, затем внешние.
  • При движении frame его unmanaged-дети получают каскадный `dx/dy`, чтобы абсолютные координаты оставались консистентными.
  • Для template/export сценариев вложенность дополнительно сохраняется в `layerTree`.
Corner cases для frame внутри frame что может удивлять
  • Self-reference: если frame случайно содержит сам себя в `childIds`, layout engine удаляет такую ссылку защитно перед расчётом.
  • Циклические связи: глубина nested frame считается с защитой от circular reference, чтобы расчёт не уходил в бесконечную рекурсию.
  • Invisible или absolute children: они не участвуют в auto-layout flow, но при движении родителя должны сохранить корректные абсолютные координаты.
  • Nested auto-layout: внутренний frame сначала может изменить собственный размер через `hug`, и только потом внешний frame измерит его как ребёнка.
  • Manual resize дочернего слоя: если слой внутри auto-layout вручную растягивается transformer’ом, `fill`/`hug` могут быть сброшены в `fixed`, чтобы не конфликтовать с ручной геометрией.
  • Text inside auto-layout: для текста `auto_width` и `auto_height` возможна повторная переоценка размеров, особенно когда ширина меняется через `fill`.
  • Drop outside frame: если слой вытаскивается полностью вне любого frame, он удаляется из родительского `childIds` и возвращается на верхний уровень.
  • Clip vs selection: слой может оставаться частью frame, но перестаёт быть кликабельным, если курсор или marquee находятся за clip bounds контейнера.
Что это значит для продукта UX-следствия
Текущая frame-модель уже достаточно сильная для template-driven composition и smart resize, но она требует хорошей UX-обвязки. Без явных подсказок пользователю может быть неочевидно, почему клик выбирает frame вместо child layer, почему часть детей не участвует в auto-layout, или почему после ручного resize поведение `fill` меняется на `fixed`. Поэтому auto-layout и nested frame здесь лучше описывать как отдельную подсистему редактора, а не как одну настройку в свойствах.

Ключевые сценарии работы

Ниже не абстрактные product flows, а те сценарии, которые действительно выстраиваются из текущего UI, store-логики и AI-интеграций.

1

С нуля в Studio

Пользователь создаёт слои через toolbar, собирает композицию во frame, редактирует свойства, получает autosave и экспортирует PNG или ZIP по resize.

2

Быстрый старт через Wizard

Шаблон выбирается заранее, dynamic fields заполняются вручную или через AI, затем pack применяется на канвас и дорабатывается в Studio.

3

Multi-format и smart resize

К проекту добавляются новые форматы, instance layout расходится по resize, а smart resize подтягивает layout из template slot mapping при сохранении текущего контента мастеров.

4

AI-генерация и AI-редактирование

Через prompt bar создаются тексты и изображения, а через image editor выбранный image layer можно доработать маской, промптом или расширением холста.

5

Чат-ассистент для канваса

AI chat строит план, предлагает style presets и templates, а затем исполняет ограниченный набор canvas actions для добавления контента и загрузки шаблонов.
