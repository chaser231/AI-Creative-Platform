## 1. Введение и цели документа

### 1.1 Назначение

Данный документ описывает **единый регламент работы с бэклогом задач** проекта AI Creative Platform. Он определяет, как задачи создаются, классифицируются, приоритизируются, планируются, выполняются и закрываются. Документ является обязательным к исполнению для всех участников команды.

### 1.2 Контекст проекта

AI Creative Platform — внутренняя платформа Яндекса для создания креативного контента с использованием AI. Проект находится в фазе **Planning / Architecture** с движением к MVP. Используется гибридная методология **Scrumban** (2-недельные спринты с Kanban-потоком для операционных задач).

### 1.3 Текущий состав команды

| Роль | Кол-во | Зона ответственности |
|------|--------|---------------------|
| Product Owner | 1 | Стратегия, приоритизация, roadmap |
| Project Manager | 1 | Координация, спринты, отчётность |
| Frontend-разработчик | 1–2 | UI, Canvas Engine, компоненты |
| Backend-разработчик | 1–2 | API, AI Orchestrator, инфраструктура |
| UX/UI-дизайнер | 1 | Дизайн-система, макеты, прототипы |
| Менеджер (операционный) | 1 | Контент, фидбек, конфиги, гайды |

### 1.4 Ключевые принципы

1. **Прозрачность** — каждый участник в любой момент видит актуальный статус всех задач.
2. **Минимум бюрократии** — процессы адаптированы под небольшую команду.
3. **Масштабируемость** — структура готова к росту команды без рефакторинга процессов.
4. **Единый источник правды** — Яндекс Трекер является единственным местом управления задачами.

## 2. Приоритеты задач и скоринг

### 2.1 Уровни приоритетов

| Приоритет | Обозначение | SLA реакции | SLA решения | Описание |
|-----------|-------------|-------------|-------------|----------|
| **Блокер** | 🔴 Blocker | 1 час | 24 часа | Полная остановка работы, критический баг в продакшене, потеря данных. Всё остальное отходит на второй план. |
| **Высокий** | 🟠 High | 4 часа | Текущий спринт | Серьёзное влияние на пользователей или блокировка других задач. Функциональность сильно деградирована. |
| **Средний** | 🟡 Medium | 1 рабочий день | 1–2 спринта | Важная задача, но есть обходной путь. Основной поток работы для плановых фич и улучшений. |
| **Низкий** | 🟢 Low | Без SLA | Backlog | Улучшения «nice-to-have», рефакторинг, технический долг, косметические правки. Берётся в работу при наличии свободных ресурсов. |

### 2.2 Модель скоринга: Адаптированный RICE

Для приоритизации задач категорий «Продуктовые», «UX» и «Технические» используется модель **RICE**, адаптированная под контекст проекта:

#### Параметры оценки

| Параметр | Описание | Шкала |
|----------|----------|-------|
| **Reach (Охват)** | Сколько пользователей затронет задача за квартал? | 1 — только admin; 3 — один сегмент (дизайнеры ИЛИ менеджеры); 5 — все пользователи |
| **Impact (Влияние)** | Насколько сильно повлияет на ключевую метрику «Creative Hours Saved»? | 0.5 — минимально; 1 — слабо; 2 — средне; 3 — сильно |
| **Confidence (Уверенность)** | Насколько мы уверены в оценках Reach и Impact? | 100% — данные подтверждены; 80% — есть гипотеза с частичной валидацией; 50% — интуиция; 20% — чистая спекуляция |
| **Effort (Трудозатраты)** | T-shirt size, приведённый к «человеко-спринтам» | S = 0.5; M = 1; L = 2; XL = 4 |

#### Формула

```
RICE Score = (Reach × Impact × Confidence) / Effort
```

#### Маппинг RICE → Приоритет

| RICE Score | Рекомендуемый приоритет |
|------------|------------------------|
| ≥ 10 | 🟠 Высокий |
| 4 – 9.9 | 🟡 Средний |
| < 4 | 🟢 Низкий |

> **Примечание:** Блокеры назначаются вне скоринга — по факту критической ситуации. RICE применяется для планирования, а не для реагирования на инциденты.

### 2.3 Принципы назначения приоритетов

1. **Блокер** назначается только при полной остановке работы или критическом баге. Решение о присвоении блокера принимает PO или PM.
2. **Высокий** — задача, без которой невозможен релиз текущего спринта или которая блокирует другие задачи в спринте.
3. **Средний** — стандартный приоритет для плановых задач, попавших в спринт по итогам RICE-скоринга.
4. **Низкий** — задачи, которые улучшают продукт, но не влияют на текущие цели спринта.
5. Приоритеты **пересматриваются** на каждом Sprint Planning и при появлении новой информации.

## 3. Категории задач

### 3.1 Обзор категорий

| Категория | Тег в Трекере | Кто отвечает | Типичный приоритет |
|-----------|---------------|--------------|-------------------|
| Операционные | `ops` | PM, Менеджер | 🟡 Средний – 🟢 Низкий |
| Технические / Разработка | `dev` | Разработчики | 🟠 Высокий – 🟡 Средний |
| UX-задачи | `ux` | Дизайнер, PO | 🟡 Средний |
| Дизайн-задачи | `design` | Дизайнер | 🟡 Средний – 🟢 Низкий |
| Продуктовые задачи | `product` | PO | 🟠 Высокий – 🟡 Средний |

### 3.2 Операционные задачи (`ops`)

**Что входит:**
- Настройка процессов и инструментов (CI/CD, мониторинг, окружения)
- Рутинные задачи (обновление зависимостей, ротация ключей)
- Сбор и обработка фидбека от пользователей
- Сбор и обновление конфигов (баннеры, шаблоны, Brand Kit)
- Подготовка гайдов, инструкций, документации
- Подготовка презентаций и демо-материалов
- Онбординг новых участников команды

**Принцип распределения:** Задача является операционной, если она **не создаёт новый продуктовый функционал**, а обеспечивает работу существующих процессов или помогает команде работать эффективнее.

**Типичный приоритет:** 🟡 Средний (рутинные — 🟢 Низкий). Исключение: настройка инфраструктуры, блокирующая разработку — 🟠 Высокий.

**Подтипы (компоненты в Трекере):**
- `ops/process` — настройка процессов
- `ops/content` — контент, конфиги, баннеры
- `ops/docs` — гайды, инструкции, презентации
- `ops/feedback` — сбор и обработка фидбека

### 3.3 Технические задачи / Разработка (`dev`)

**Что входит:**
- **Frontend:** UI-компоненты, Canvas Engine, интеграция с дизайн-системой, клиентская логика
- **Backend:** API, база данных, AI Orchestrator, микросервисы, инфраструктура
- **Общее:** Баг-фиксы, рефакторинг, перфоманс-оптимизация, техдолг, тесты

**Принцип распределения:** Задача является технической, если она требует **написания или изменения кода** (фронтенд или бэкенд) и напрямую влияет на функциональность платформы.

**Типичный приоритет:** Определяется RICE-скором. Баг-фиксы: от 🔴 Блокер до 🟢 Низкий в зависимости от severity.

**Подтипы (компоненты в Трекере):**
- `dev/frontend` — фронтенд-разработка
- `dev/backend` — бэкенд-разработка
- `dev/infra` — инфраструктура, DevOps
- `dev/bugfix` — исправление ошибок
- `dev/techdebt` — технический долг

### 3.4 UX-задачи (`ux`)

**Что входит:**
- Улучшение текущих пользовательских сценариев (на основе данных и фидбека)
- Проработка новых сценариев и флоу (Wizard Mode, Studio Mode)
- UX-гипотезы и эксперименты
- Юзабилити-тестирование и анализ результатов
- Проработка информационной архитектуры

**Принцип распределения:** Задача является UX-задачей, если она направлена на **исследование, проектирование или валидацию пользовательского опыта**, но ещё не перешла в фазу визуального дизайна или разработки.

**Типичный приоритет:** 🟡 Средний. UX-задачи, связанные с валидацией гипотез для ближайших фич — 🟠 Высокий.

**Подтипы:**
- `ux/improvement` — улучшение существующего
- `ux/new` — проработка нового
- `ux/hypothesis` — гипотезы и эксперименты
- `ux/research` — исследования (интервью, тесты)

### 3.5 Дизайн-задачи (`design`)

**Что входит:**
- Разработка и развитие дизайн-системы (токены, компоненты, паттерны)
- Создание макетов и прототипов для конкретных экранов
- Иконки, иллюстрации, визуальные ассеты
- Адаптация макетов под разные темы (Light/Dark) и форм-факторы
- Ревью визуальной реализации (Design QA)

**Принцип распределения:** Задача является дизайн-задачей, если её результатом является **визуальный артефакт** (макет, компонент дизайн-системы, ассет), а не исследование или код.

**Типичный приоритет:** 🟡 Средний для плановых задач; 🟠 Высокий если дизайн блокирует разработку.

**Подтипы:**
- `design/system` — дизайн-система
- `design/mockup` — макеты и прототипы
- `design/asset` — визуальные ассеты
- `design/qa` — Design QA

### 3.6 Продуктовые задачи (`product`)

**Что входит:**
- Формирование и обновление Product Roadmap
- Описание фич и Epic-ов (PRD — Product Requirements Document)
- Discovery: формулировка проблем, генерация гипотез, валидация
- Анализ метрик и принятие решений (Build / Pivot / Kill)
- Стратегические решения (какую AI-модель интегрировать, какой сегмент обслужить первым)
- Приоритизация бэклога (RICE-сессии)

**Принцип распределения:** Задача является продуктовой, если она определяет **«что» и «зачем»** мы делаем, а не «как» (это UX/дизайн/разработка).

**Типичный приоритет:** 🟠 Высокий для стратегических решений, блокирующих планирование; 🟡 Средний для текущей аналитики.

**Подтипы:**
- `product/strategy` — стратегия и roadmap
- `product/discovery` — исследование и валидация
- `product/prd` — описание требований
- `product/analytics` — метрики и аналитика

### 3.7 Матрица: категория × приоритет (типичные кейсы)

| Ситуация | Категория | Приоритет |
|----------|-----------|-----------|
| Продакшен упал, пользователи не могут работать | `dev/bugfix` | 🔴 Блокер |
| Баг в рендеринге Canvas, данные не теряются, есть workaround | `dev/bugfix` | 🟠 Высокий |
| Новая фича «Smart Resize» — ядро следующего спринта | `dev/frontend` | 🟠 Высокий |
| Рефакторинг API-клиента | `dev/techdebt` | 🟢 Низкий |
| Макеты для «Smart Resize» нужны до начала разработки | `design/mockup` | 🟠 Высокий |
| Обновление иконок в дизайн-системе | `design/asset` | 🟢 Низкий |
| UX-тест гипотезы «Wizard Mode ускоряет онбординг» | `ux/hypothesis` | 🟡 Средний |
| Написать гайд по экспорту баннеров | `ops/docs` | 🟢 Низкий |
| Собрать фидбек по бета-версии от 5 дизайнеров | `ops/feedback` | 🟡 Средний |
| PRD для Phase 2 «Design Engine» | `product/prd` | 🟠 Высокий |

## 4. Воркфлоу и жизненный цикл задач

### 4.1 Основной воркфлоу (для всех категорий)

```
┌─────────┐    ┌────────────┐    ┌─────────────┐    ┌───────────┐    ┌──────────┐    ┌────────┐
│  Open   │───▸│ In Planning│───▸│ In Progress │───▸│ In Review │───▸│  Testing │───▸│ Closed │
└─────────┘    └────────────┘    └─────────────┘    └───────────┘    └──────────┘    └────────┘
     │                                   │                │                │
     │                                   ▼                ▼                ▼
     │                            ┌────────────┐   ┌────────────┐   ┌────────────┐
     └───────────────────────────▸│ On Hold    │   │ Rejected   │   │ Reopened   │
                                  └────────────┘   └────────────┘   └────────────┘
```

### 4.2 Статусы и их описание

| Статус | Описание | Кто переводит |
|--------|----------|---------------|
| **Open** | Задача создана, ждёт приоритизации и оценки | Автор задачи |
| **In Planning** | Задача оценена, декомпозирована, готова к спринту | PO / PM |
| **In Progress** | Исполнитель взял задачу в работу | Исполнитель |
| **In Review** | Задача на ревью (Code Review / Design Review / PO Review) | Исполнитель |
| **Testing** | Задача на тестировании (QA или автотесты) | Ревьюер |
| **Closed** | Задача выполнена и принята | PM / PO |
| **On Hold** | Задача заблокирована (ждёт зависимость, решение, ресурсы) | Любой (с комментарием) |
| **Rejected** | Задача отклонена (неактуальна, дубликат) | PO |
| **Reopened** | Задача требует доработки после ревью/тестирования | Ревьюер / QA |

### 4.3 Воркфлоу по категориям

#### Технические задачи (`dev`)

```
Open → In Planning → In Progress → In Review (Code Review) → Testing → Closed
```

- **In Review** = обязательный Code Review через Pull Request.
- **Testing** = автотесты + ручная проверка в Staging-окружении.
- Для `dev/bugfix` с приоритетом 🔴 Блокер допускается упрощённый флоу: `Open → In Progress → In Review → Closed` (hotfix).

#### UX-задачи (`ux`)

```
Open → In Planning → In Progress → In Review (PO Review) → Closed
```

- **In Review** = ревью от PO (валидация результатов исследования/прототипа).
- **Testing** не обязателен (UX-задачи не порождают код напрямую).
- Результат UX-задачи часто порождает дочерние задачи `design` и `dev`.

#### Дизайн-задачи (`design`)

```
Open → In Planning → In Progress → In Review (Design Review) → Closed
```

- **In Review** = ревью от PO + ведущего дизайнера.
- Для `design/qa` флоу: `Open → In Progress → Closed/Reopened` (ревью реализации).

#### Операционные задачи (`ops`)

```
Open → In Progress → Closed
```

- Упрощённый флоу без обязательного ревью.
- Для `ops/process` (настройка CI/CD и пр.) используется полный флоу с ревью.

#### Продуктовые задачи (`product`)

```
Open → In Planning → In Progress → In Review (Stakeholder Review) → Closed
```

- **In Review** = согласование с заинтересованными сторонами.
- Результат продуктовой задачи = документ (PRD, стратегия), который становится входом для UX/дизайн/dev задач.

### 4.4 Связи между задачами

Используем типы связей Яндекс Трекера:

| Тип связи | Когда использовать | Пример |
|-----------|--------------------|--------|
| **Parent → Sub-task** | Декомпозиция Epic → Story → Task | Epic «Smart Resize» → Story «UI ресайза» → Task «Компонент дропдауна» |
| **Depends on** | Задача A не может быть начата без завершения B | `dev/frontend`: «UI ресайза» depends on `design/mockup`: «Макет ресайза» |
| **Related** | Задачи логически связаны, но не блокируют друг друга | `ux/research`: «Тест Wizard Mode» related to `product/discovery`: «Гипотеза онбординга» |

## 5. Планирование и распределение нагрузки

### 5.1 Ритм и церемонии

Проект работает в **2-недельных спринтах** (Scrumban). Церемонии:

| Церемония | Когда | Длительность | Участники | Цель |
|-----------|-------|--------------|-----------|------|
| **Sprint Planning** | Пн, 1-й день спринта | 1.5 часа | Вся команда | Выбрать задачи на спринт, оценить, распределить |
| **Daily Standup** | Ежедневно (кроме Пн 1-го дня) | 15 минут | Вся команда | Синхронизация: что сделал, что буду делать, блокеры |
| **Backlog Grooming** | Ср, 1-я неделя спринта | 1 час | PO, PM, тех-лиды | Уточнение задач, RICE-скоринг, декомпозиция |
| **Sprint Review / Demo** | Пт, последний день спринта | 1 час | Вся команда + стейкхолдеры | Демо результатов спринта |
| **Retrospective** | Пт, после Demo | 45 минут | Вся команда | Что было хорошо, что улучшить |

### 5.2 Принципы планирования спринта

1. **Capacity planning:** Каждый участник имеет бюджет **8 story points** на спринт (при полной загрузке). Планируем на **80%** ёмкости (≈6.5 SP), оставляя буфер на непредвиденные задачи.
2. **Обязательный микс:** В каждом спринте должны быть:
   - Минимум **60–70%** ёмкости на задачи 🟠 Высокого и 🟡 Среднего приоритета (фичи, UX, дизайн).
   - До **20%** на техдолг и рефакторинг (`dev/techdebt`).
   - До **10%** на операционные задачи (`ops`).
3. **WIP-лимиты (Work In Progress):**
   - Разработчик: не более **2 задач** одновременно в статусе `In Progress`.
   - Дизайнер: не более **3 задач** одновременно.
   - Это контролируется через настройки доски в Трекере.
4. **Правило «Если не в спринте — не в работе»**: Задачи, не включённые в текущий спринт, не берутся в работу (исключение — 🔴 Блокеры).

### 5.3 Оценка задач: Покер планирования

**Шкала Story Points:** 1, 2, 3, 5, 8, 13 (модифицированный Фибоначчи).

| SP | Калибровка | Пример |
|----|------------|--------|
| 1 | Менее 2 часов, тривиальная задача | Исправить текст на кнопке |
| 2 | Полдня, понятная задача | Добавить новое поле в форму |
| 3 | 1 день, средняя сложность | Реализовать компонент карточки ассета |
| 5 | 2–3 дня, есть неопределённость | Интеграция с AI Orchestrator для одного use case |
| 8 | Почти весь спринт, высокая сложность | Реализация Canvas Engine: Zoom + Pan |
| 13 | Слишком большая, **требует декомпозиции** | Epic «Smart Resize» → разбить на Stories |

**Процесс покера:**
1. PO зачитывает задачу и acceptance criteria.
2. Участники выбирают оценку (в Трекере — через функцию «Покер планирования» на доске).
3. Если оценки расходятся более чем на 2 уровня, крайние объясняют позицию.
4. Повторное голосование → берём консенсус.
5. Оценка записывается в поле `Story Points` задачи.

### 5.4 Распределение задач

| Принцип | Описание |
|---------|----------|
| **Ownership** | Каждая задача имеет одного исполнителя (поле `Assignee`). |
| **Self-assignment** | На Sprint Planning задачи распределяются совместно, но исполнитель сам «берёт» задачу, а не получает назначение сверху. |
| **T-shape** | Каждый участник имеет основную специализацию, но может помочь в смежных областях при необходимости. |
| **Парная работа** | Для задач с оценкой ≥8 SP или высокой неопределённостью рекомендуется назначить пару (основной + ревьюер). |
| **Дизайн → Разработка pipeline** | Дизайн-задачи планируются на **1 спринт раньше** соответствующих dev-задач. |

### 6.2 Процесс заведения задачи

#### Шаг 1: Создание задачи

Любой участник команды может создать задачу в Трекере. Обязательные поля при создании:

| Поле | Обязательно | Описание |
|------|-------------|----------|
| **Название** | ✅ | Краткое, действие-ориентированное. Пример: «Реализовать компонент Smart Resize» |
| **Описание** | ✅ | Контекст, acceptance criteria, ссылки на макеты/PRD |
| **Категория (тег)** | ✅ | Один из: `ops`, `dev`, `ux`, `design`, `product` |
| **Подтип (компонент)** | ✅ | Например: `dev/frontend`, `ux/hypothesis` |
| **Приоритет** | ⚠️ Предварительный | Автор ставит предварительный приоритет; финальный — на Grooming |
| **Story Points** | ❌ | Оцениваются на Sprint Planning (покер) |
| **Assignee** | ❌ | Назначается на Sprint Planning |

#### Шаг 2: Triage (сортировка)

- **Кто:** PO или PM.
- **Когда:** Ежедневно (или в рамках Grooming).
- **Что делает:**
  1. Проверяет корректность категории и подтипа.
  2. Проверяет / уточняет приоритет.
  3. Проверяет наличие достаточного описания.
  4. Связывает с родительским Epic-ом (если применимо).
  5. Переводит статус из `Open` → `In Planning` (готово к оценке).

#### Шаг 3: Grooming

- Задача обсуждается на Backlog Grooming.
- Проводится RICE-скоринг (для product / ux / dev задач).
- Задача декомпозируется, если слишком крупная.

#### Шаг 4: Sprint Planning

- Задача оценивается покером (Story Points).
- Назначается исполнитель.
- Включается в спринт.

## 7. Настройка Яндекс Трекера

### 7.1 Структура очередей

Для проекта создаём **одну основную очередь** с разделением по компонентам (а не несколько очередей). Это оптимально для небольшой команды и упрощает кросс-функциональное взаимодействие.

| Элемент | Значение | Обоснование |
|---------|----------|-------------|
| **Очередь** | `ACP` (AI Creative Platform) | Единая очередь = единый контекст. Легко фильтровать. |
| **Ключ задачи** | `ACP-123` | Единый формат для всех задач проекта. |

#### Компоненты очереди (для фильтрации)

Настроить в Трекере: **Очередь → Настройки → Компоненты**

```
ops/process
ops/content
ops/docs
ops/feedback
dev/frontend
dev/backend
dev/infra
dev/bugfix
dev/techdebt
ux/improvement
ux/new
ux/hypothesis
ux/research
design/system
design/mockup
design/asset
design/qa
product/strategy
product/discovery
product/prd
product/analytics
```

#### Типы задач

Настроить в Трекере: **Очередь → Настройки → Типы задач**

| Тип | Когда использовать |
|-----|-------------------|
| **Epic** | Крупная фича или инициатива (Phase 1: MVP, Smart Resize, AI Layer) |
| **Story** | Пользовательская история или крупный блок работ внутри Epic |
| **Task** | Конкретная задача для исполнителя |
| **Bug** | Дефект, ошибка |
| **Sub-task** | Подзадача внутри Task или Story |

#### Кастомные поля

Настроить в Трекере: **Очередь → Настройки → Поля задачи**

| Поле | Тип | Значения | Назначение |
|------|-----|----------|------------|
| `Категория` | Список | ops, dev, ux, design, product | Основная категория задачи |
| `RICE: Reach` | Число | 1–5 | RICE-скоринг |
| `RICE: Impact` | Число (дробное) | 0.5, 1, 2, 3 | RICE-скоринг |
| `RICE: Confidence` | Число | 20, 50, 80, 100 | RICE-скоринг |
| `RICE: Effort` | Список | S(0.5), M(1), L(2), XL(4) | RICE-скоринг |
| `RICE Score` | Вычисляемое(формула) | — | Автоматически считается |
| `Story Points` | Число | 1, 2, 3, 5, 8, 13 | Оценка трудоёмкости |
| `Phase` | Список | 1, 2, 3, 4 | Привязка к фазе развития продукта |

### 7.2 Настройка досок

Создаём **3 доски** с разными целями для разных ролей:

#### Доска 1: «Sprint Board» (основная)

**Тип:** Доска с бэклогом и спринтами  
**Для кого:** Вся команда (ежедневное использование)  
**Фильтр:** Очередь = `ACP`

**Колонки:**

| Колонка | Маппинг на статус | WIP-лимит |
|---------|-------------------|-----------|
| Backlog | Open, In Planning | — |
| To Do | In Planning (добавлено в спринт) | — |
| In Progress | In Progress | 4 (для всей команды) |
| In Review | In Review | 3 |
| Testing | Testing | 3 |
| Done | Closed | — |

**Настройка спринтов:**
- Длительность: 2 недели.
- Автозавершение: включено (незакрытые задачи переносятся в новый спринт).
- Цель спринта: заполнять в поле описания спринта.

**Группировка карточек:** По полю `Assignee` (удобно на стендапах).

**Цвет карточек** (по приоритету):
- 🔴 Блокер — красный
- 🟠 Высокий — оранжевый
- 🟡 Средний — жёлтый (по умолчанию)
- 🟢 Низкий — зелёный

#### Доска 2: «Kanban Ops» (операционный поток)

**Тип:** Простая доска  
**Для кого:** PM, менеджер  
**Фильтр:** Очередь = `ACP` И Категория = `ops`

**Колонки:**

| Колонка | Маппинг на статус |
|---------|-------------------|
| Inbox | Open |
| In Progress | In Progress |
| Done | Closed |

**Обоснование:** Операционные задачи часто не привязаны к спринтам. Kanban-поток с простыми колонками позволяет менеджеру видеть свой пул задач без «шума» от dev/ux/design задач.

#### Доска 3: «Product & UX Pipeline»

**Тип:** Простая доска  
**Для кого:** PO, дизайнер  
**Фильтр:** Очередь = `ACP` И Категория IN (`product`, `ux`, `design`)

**Колонки:**

| Колонка | Маппинг на статус |
|---------|-------------------|
| Ideas | Open |
| In Planning | In Planning |
| In Work | In Progress |
| Review | In Review |
| Done | Closed |

**Обоснование:** PO и дизайнер видят полный pipeline от идеи до реализации для продуктовых, UX и дизайн задач. Удобно для отслеживания «конвейера фич».

### 7.3 Диаграмма Ганта

**Где:** Проекты → AI Creative Platform → Диаграмма Ганта

**Для кого:** PO, PM (долгосрочное планирование)

**Настройка:**
1. Создать **Проект** «AI Creative Platform» в Трекере.
2. Привязать все Epic-задачи к проекту.
3. Для каждого Epic задать **даты начала и окончания** (в полях «Дата начала» и «Дедлайн»).
4. Настроить **зависимости между Epic-ами** (Phase 1 → Phase 2 → Phase 3 → Phase 4).
5. Гант автоматически покажет визуальную шкалу времени со связями.

**Рекомендации:**
- Показывать на Ганте только уровень **Epic** и **Story** (не задачи — слишком гранулярно).
- Использовать Гант для **квартального планирования** и согласования со стейкхолдерами.
- Не использовать Гант для ежедневного управления — для этого есть Sprint Board.

### 7.4 Покер планирования

**Где:** Sprint Board → Покер планирования (кнопка на панели спринта)

**Когда:** На Sprint Planning (каждые 2 недели)

**Настройка:**
1. Открыть Sprint Board.
2. Выбрать задачи, которые будут оцениваться в текущем спринте.
3. Запустить «Покер планирования» через интерфейс доски.
4. Шкала оценок: Fibonacci (1, 2, 3, 5, 8, 13, 21, ?).
5. Каждый участник голосует → обсуждение при расхождении → консенсус.
6. Оценка автоматически записывается в поле `Story Points` задачи.

### 7.5 Фильтры и дашборды

#### Сохранённые фильтры (создать для быстрого доступа)

| Название фильтра | Запрос (язык запросов Трекера) | Для кого |
|-------------------|-------------------------------|----------|
| «Мои задачи» | `Queue: ACP Assignee: me() Status: !Closed` | Каждый участник |
| «Блокеры» | `Queue: ACP Priority: Blocker Status: !Closed` | Вся команда |
| «Текущий спринт» | `Queue: ACP Sprint: currentSprint()` | Вся команда |
| «Незаоценённые» | `Queue: ACP "Story Points": empty() Status: "In Planning"` | PM, на Grooming |
| «Бэклог Frontend» | `Queue: ACP Components: "dev/frontend" Status: Open, "In Planning"` | Frontend-разработчик |
| «Бэклог Backend» | `Queue: ACP Components: "dev/backend" Status: Open, "In Planning"` | Backend-разработчик |

| Задачи по категориям | Диаграмма | Распределение задач по полю «Категория» |
| Блокеры | Список задач | Фильтр: все 🔴 Блокеры |

## 9. Масштабирование

### 9.1 Текущая структура (3–7 человек)

```
Одна очередь ACP → Одна Sprint Board → 2 дополнительные доски
```

### 9.2 При росте до 8–15 человек

| Изменение | Что делать |
|-----------|------------|
| **Создать под-очереди** | `ACP-FE` (Frontend), `ACP-BE` (Backend), `ACP-DESIGN` — отдельные очереди для специализированных команд |
| **Feature Teams** | Перейти от функциональных команд к feature-командам (каждая реализует фичу от дизайна до деплоя) |
| **Scrum of Scrums** | Еженедельная синхронизация между лидами feature-команд |
| **Agile Board per Team** | У каждой feature-команды — своя Sprint Board |
| **Общая Product Board** | Доска уровня Epic для PO и стейкхолдеров |
| **Портфель проектов** | Использовать «Портфели» в Трекере для агрегированного view |

### 9.3 При росте до 15+ человек

- Ввести роль **Scrum Master** (отдельно от PM).
- Использовать **«Цели»** в Трекере для каскадирования OKR → Epic → Story.
- Рассмотреть **SAFe** (Scaled Agile Framework) элементы: Program Increment Planning, PI-board.
- Перейти к **квартальному планированию** с фиксированными PI (Program Increment).

### 9.4 Чеклист готовности к масштабированию

- [ ] Все процессы задокументированы в данном регламенте и актуализированы.
- [ ] Каждая категория задач имеет владельца (Lead).
- [ ] Velocity стабильна на протяжении 5+ спринтов.
- [ ] Автоматизация тригеров в Трекере настроена и работает.
- [ ] Онбординг нового участника занимает не более 2 дней.
- [ ] Дашборд покрывает все ключевые метрики.

## Приложение B: Глоссарий

| Термин | Определение |
|--------|------------|
| **Epic** | Крупная инициатива, объединяющая несколько Stories (пример: «Phase 1: MVP») |
| **Story** | Пользовательская история или крупный блок работ внутри Epic |
| **Task** | Конкретная задача для одного исполнителя |
| **Story Points (SP)** | Абстрактная единица оценки сложности задачи |
| **RICE** | Метод приоритизации: Reach × Impact × Confidence / Effort |
| **WIP** | Work In Progress — количество задач в одновременной работе |
| **Velocity** | Количество SP, закрываемых командой за спринт (средняя за 3–5 спринтов) |
| **Scrumban** | Гибрид Scrum (спринты) и Kanban (поток, WIP-лимиты) |
| **Triage** | Процесс первичной сортировки и классификации новых задач |
| **Grooming (Refinement)** | Уточнение задач бэклога: описание, декомпозиция, оценка |
| **Burndown Chart** | График сгорания SP в спринте (план vs факт) |
